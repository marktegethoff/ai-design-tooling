# Onboarding

**Eyebrow:** Pillar 04

**Subtitle:** New designers should start within the team's behavioral contract — not from a blank CLI. Context arrives first. Tools operate within it.

---

> **Warning: Self-service terminal install fails for non-technical designers.**
> A "clone the repo and run one command" onboarding only works if every prerequisite is already in place. At Capital One, they won't be. The install must be owned by IT — pre-deployed as part of the standard design tooling package via MDM. Designers should never touch the command line for setup.

---

## What the onboarding package contains

**Team CLAUDE.md**
Pre-populated with team standards, design system, decision hierarchy, and org layer import. Ready to use on day one.

**Discipline agents**
The team's agent set installed to `~/.claude/agents/`. New members get the same specialists as everyone else from first session.

**Workflow skills**
Team-validated workflow skills installed and ready. The right sequence for design, review, and implementation is already encoded.

---

## The first session experience

The designer's first Claude Code session should demonstrate the behavioral contract working — not require them to build it. Recommended first session structure:

1. **Show the team CLAUDE.md** — Read it together. Explain what each section governs and why it's there.
2. **Make one edit, see the consequence** — Add a rule. Run a session. Show how Claude's behavior changed. This is the behavioral contract demonstration.
3. **Run one real task** — A real task from their actual work, not a tutorial exercise. The first session should produce something they can use.
4. **Add one personal context entry** — What are you working on? What decisions have you made? Their first memory entry, written together.
