## How the system is structured

Five things our team needs to make this work. Each one covers a different need. None overlaps. All five are in place here.

**01 — Capability Proof**
A real demonstration of what one designer can build with this system. Creates pull. Senior designers respond to proof, not process documentation.

**02 — Vocabulary**
Calling a CLAUDE.md a "behavioral contract" — not a config file — changes how designers relate to maintaining it. Vocabulary determines behavior.

**03 — Governance**
A three-layer inheritance structure — org → team → personal — so shared context is always present without overriding team autonomy.

**04 — Onboarding**
New designers start within the team's behavioral contract, not a blank CLI. Context arrives first; tools operate within it.

**05 — Commit Ceremony**
One sentence at the end of every team session. Three months of that and you have institutional memory the AI can use.

## We start with proof, not setup

The sequence matters. If you come to us asking for the tool setup first, we'll show you the capability proof first. That's intentional — the system doesn't land until you've seen what it makes possible.
